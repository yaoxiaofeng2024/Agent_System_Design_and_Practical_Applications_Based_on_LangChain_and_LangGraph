# 智能体实战之RAG深入：从向量数据库到检索优化的全链路实战

## 一.简介

第14章介绍了RAG的核心原理与开发流程，并引用了外部项目实现PDF多模态RAG。本章将深入RAG的工程实践细节，补全从文档处理到检索优化的完整技术链路，这些是构建生产级RAG系统的必备知识。

### 1. 向量数据库：RAG的存储引擎

向量数据库是RAG系统的核心基础设施。它负责存储文档的向量表示，并支持高效的相似性搜索。常见的向量数据库包括：

* **FAISS**：Facebook开源的高效向量搜索库，适合单机场景，性能极佳，但不支持数据持久化和分布式。
* **Chroma**：轻量级开源向量数据库，内置嵌入支持，适合开发和小规模部署。
* **Milvus**：分布式向量数据库，支持亿级向量搜索，适合企业级大规模部署。
* **Pinecone**：全托管向量数据库服务，零运维，适合快速上线。

选择向量数据库时需要考虑：数据规模、查询延迟、是否需要持久化、部署方式（本地/云端）以及与LangChain的集成程度。

### 2. 文档分块策略：检索质量的基石

分块（Chunking）是RAG流程中最容易被忽视但影响最大的环节。分块太大，检索到的内容包含过多无关信息，干扰LLM生成；分块太小，丢失上下文，导致答案不完整。

常见分块策略：

* **固定大小分块**：按字符数或Token数切分，简单但可能切断语义。
* **递归字符分块**：按分隔符层级（段落→句子→字符）递归切分，尽量保持语义完整。
* **语义分块**：利用嵌入模型计算相邻句子的语义相似度，在语义断裂处切分，质量最高但成本较大。
* **文档结构分块**：按Markdown标题、HTML标签等文档结构切分，保留逻辑层次。

### 3. 嵌入模型：语义理解的核心

嵌入模型将文本转换为高维向量，其质量直接决定检索的准确性。

* **OpenAI Embeddings**：`text-embedding-3-small/large`，质量高但需付费。
* **BGE系列**：BAAI开源的中文嵌入模型，`bge-small-zh-v1.5`轻量高效，`bge-large-zh-v1.5`精度更高。
* **GTE系列**：阿里开源的多语言嵌入模型，中文表现优秀。
* **Cohere Embeddings**：多语言支持好，有免费额度。

选择嵌入模型时需权衡：向量维度（影响存储和搜索速度）、多语言支持、推理速度和准确率。

### 4. 检索优化：从简单到高级

基础RAG使用单一的向量相似性搜索，但在实际场景中往往不够：

* **混合检索（Hybrid Search）**：结合向量搜索（语义匹配）和关键词搜索（精确匹配），取长补短。向量搜索擅长理解意图，关键词搜索擅长精确匹配专有名词、编号等。
* **重排序（Reranking）**：先用向量搜索召回大量候选（如Top-20），再用交叉编码器（Cross-Encoder）对候选进行精排，取Top-K（如Top-5）。交叉编码器同时考虑查询和文档的交互，精度远高于双塔模型。
* **查询改写（Query Rewriting）**：让LLM将用户的模糊查询改写为更精确的检索查询，或生成多个子查询分别检索后合并。
* **上下文压缩（Contextual Compression）**：对检索到的长文档进行摘要或提取与查询相关的片段，减少传递给LLM的无关信息。

### 5. RAG评估：量化系统质量

没有评估就无法优化。RAG系统需要从两个维度评估：

* **检索质量**：检索到的文档是否相关？覆盖率如何？
* **生成质量**：最终答案是否准确、完整、有依据？

专业评估框架如RAGAS提供了标准化的评估指标，包括：忠实度（Faithfulness）、答案相关性（Answer Relevance）、上下文精确率（Context Precision）、上下文召回率（Context Recall）等。

## 二.案例：企业技术文档知识库问答系统

构建一个企业技术文档知识库问答系统，覆盖RAG的全链路工程实践：

1. **文档加载与分块**：加载技术文档，使用递归字符分块策略切分
2. **向量化与存储**：使用HuggingFace嵌入模型将文本块向量化，存储到FAISS向量数据库
3. **混合检索**：结合向量搜索和关键词搜索，提高检索召回率
4. **重排序**：对检索结果进行重排序，提升精度
5. **增强生成**：将检索结果注入提示词，生成准确答案

for q in questions:

    print(f"问题: {q}")
    print(f"{'='*60}")

    )
    chain = rewrite_prompt | llm | StrOutputParser()
    rewritten = chain.invoke({"question": state["question"]})
    print(f"  原始问题: {state['question']}")
    print(f"  改写查询: {rewritten}")
    return {"rewritten_query": rewritten}

def retrieve_node(state: RAGState):
    """检索节点：从向量数据库中检索相关文档"""
    print("--- 节点2: 文档检索 ---")
    query = state.get("rewritten_query", state["question"])
    retriever = vectorstore.as_retriever(
        search_type="mmr",
        search_kwargs={"k": 4, "fetch_k": 10}
    )
    docs = retriever.invoke(query)
    print(f"  检索到 {len(docs)} 个文档块")
    return {"documents": docs}

def format_context_node(state: RAGState):
    """上下文格式化节点"""
    print("--- 节点3: 上下文格式化 ---")
    context = "\n\n---\n\n".join(
        f"[来源: {doc.metadata.get('source', '未知')}, 章节: {doc.metadata.get('section', '未知')}]\n{doc.page_content}"
        for doc in state["documents"]
    )
    return {"context": context}

def generate_node(state: RAGState):
    """生成节点：基于检索结果生成答案"""
    print("--- 节点4: 答案生成 ---")
    rag_prompt = ChatPromptTemplate.from_template(
        "你是一位专业的技术架构师。请根据以下参考文档回答用户的问题。\n\n"
        "参考文档：\n{context}\n\n"
        "用户问题：{question}\n\n"
        "要求：\n"
        "1. 只基于参考文档中的信息回答，不要编造\n"
        "2. 如果参考文档中没有相关信息，请明确说明\n"
        "3. 引用具体的文档来源\n"
        "回答："
    )
    chain = rag_prompt | llm | StrOutputParser()
    answer = chain.invoke({
        "context": state["context"],
        "question": state["question"]
    })
    return {"answer": answer}

def evaluate_node(state: RAGState):
    """评估节点：检查答案质量，决定是否需要重试"""
    print("--- 节点5: 答案评估 ---")
    if state["retry_count"] >= 1:
        print("  已达到最大重试次数，返回当前答案")
        return {"need_retry": False}

    eval_prompt = ChatPromptTemplate.from_template(
        "请评估以下RAG系统的回答质量。如果回答中包含'没有相关信息'或'无法回答'，" +
        "且这可能是因为检索不够充分，请回复'RETRY'。否则回复'PASS'。\n\n" +
        "问题：{question}\n回答：{answer}\n\n评估结果："
    )
    chain = eval_prompt | llm | StrOutputParser()
    result = chain.invoke({
        "question": state["question"],
        "answer": state["answer"]
    }).strip()
    need_retry = "RETRY" in result
    print(f"  评估结果: {result}, 需要重试: {need_retry}")
    return {"need_retry": need_retry, "retry_count": state["retry_count"] + 1}

# --- 4. 构建图 ---
def should_retry(state: RAGState) -> str:
    if state["need_retry"]:
        return "rewrite_query"
    return END

workflow = StateGraph(RAGState)

workflow.add_node("rewrite_query", rewrite_query_node)
workflow.add_node("retrieve", retrieve_node)
workflow.add_node("format_context", format_context_node)
workflow.add_node("generate", generate_node)
workflow.add_node("evaluate", evaluate_node)

workflow.add_edge(START, "rewrite_query")
workflow.add_edge("rewrite_query", "retrieve")
workflow.add_edge("retrieve", "format_context")
workflow.add_edge("format_context", "generate")
workflow.add_edge("generate", "evaluate")
workflow.add_conditional_edges("evaluate", should_retry, {"rewrite_query": "rewrite_query", END: END})

app = workflow.compile()

# --- 5. 运行 ---
questions = [
    "API网关有哪些选择？怎么选型？",
    "缓存穿透、击穿和雪崩分别是什么？怎么解决？",
    "消息队列怎么选型？",
]

for q in questions:
    print(f"\n{'='*60}")
    print(f"问题: {q}")
    print(f"{'='*60}")
    initial_state = {
        "question": q,
        "rewritten_query": "",
        "documents": [],
        "context": "",
        "answer": "",
        "need_retry": False,
        "retry_count": 0
    }
    final_state = app.invoke(initial_state)

## 五.关键知识点总结

### 分块策略选择指南

| 策略 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| 固定大小 | 简单文档、快速原型 | 实现简单 | 可能切断语义 |
| 递归字符 | 通用场景（推荐默认） | 平衡效果与效率 | 不理解文档结构 |
| 语义分块 | 高质量要求场景 | 保持语义完整 | 计算成本高 |
| 文档结构 | Markdown/HTML文档 | 保留逻辑层次 | 依赖文档格式 |

### 向量数据库选择指南

| 数据库 | 适用场景 | 持久化 | 分布式 | 部署方式 |
|--------|---------|--------|--------|---------|
| FAISS | 开发测试、小规模 | 需手动 | 不支持 | 本地 |
| Chroma | 开发测试、小规模 | 支持 | 不支持 | 本地/云端 |
| Milvus | 企业级大规模 | 支持 | 支持 | 自部署/云端 |
| Pinecone | 快速上线 | 支持 | 支持 | 全托管 |

### 检索优化策略对比

| 策略 | 效果 | 成本 | 实现复杂度 | 推荐场景 |
|------|------|------|-----------|---------|
| 向量搜索 | 基础 | 低 | 低 | 语义匹配场景 |
| 混合检索 | 较好 | 中 | 中 | 通用场景 |
| 重排序 | 好 | 较高 | 中 | 精度要求高 |
| 查询改写 | 较好 | 中 | 中 | 用户查询模糊 |
| 上下文压缩 | 好 | 中 | 较高 | 长文档场景 |
